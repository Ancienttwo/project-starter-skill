## 一、可组合性定义

### 1.1 什么是可组合性

> "每个资金盘设计都基于封闭系统模型，该模型有受众、适用条件和承受极限。超过极限将导致崩盘。为防止这种情况，设计资金盘时需要嵌套不同类型的模型，以最大程度扩展适用范围。"

### 1.2 传统vs Crypto

**传统资金盘**：称为"平移"
- 开发结构关系，很难在原有盘子应用上一边使用一边大改

**Crypto**：称为"可组合性"
- 由于开放式协议的设计
- 可以在不影响当前盘面的情况下
- 将新的盘子模块嵌入已有产品中
- 甚至直接将其他盘子的一部分嵌套进来

### 1.3 可组合性的意义

通过可组合性，可以实现：
- 三种盘型的优势互补
- 突破单一盘型的承受极限
- 最大程度扩展适用范围

## 二、三盘各自的角色

### 2.1 分红盘的角色：锁

**核心功能**：锁定用户资金

**实现方式**：
- 矿机购买（沉没成本）
- 质押锁仓
- 分红周期

**在组合中的作用**：
- 减少抛压
- 形成质押需求
- 提供稳定的资金池

### 2.2 互助盘的角色：快

**核心功能**：快速扩大规模

**实现方式**：
- 资金期限错配
- 高APY吸引
- 快速流转

**在组合中的作用**：
- 快速汇聚资金
- 抽水获利
- 提供流动性规模

### 2.3 拆分盘的角色：销

**核心功能**：创造增值预期

**实现方式**：
- 不断产生新标的
- 资产增值逻辑
- 退出流动性

**在组合中的作用**：
- 吸引增量资金
- 提供价格上涨预期
- 通过AMM提供退出通道

## 三、典型案例：Restaking

### 3.1 Restaking的本质

> "Restaking是三盘结合中'可组合性'的代表，因为他本身是整个公链的起盘组件。通过Restaking，公链可以整合不同的资金盘项目"

### 3.2 Restaking的互助盘本质

**定义**：
> "Restaking的本质是以太坊质押的'P2P'，用户通过质押ETH或LSD赚取租金，同时享受LSD和再质押的收益"

**泡沫来源**：
> "Restaking是一个典型的互助盘，它的泡沫源自于被质押的ETH的机会成本"

### 3.3 杠杆问题

**朴素逻辑**：
> "如果ETH质押了LSD，LSD又质押了LRT，当LSD节点和LRT质押节点同时因为作恶被扣钱，一份钱怎么被两份扣？这其实就是杠杆。"

> "这里存在资金错配"

### 3.4 三盘元素分解

| 盘型 | 在Restaking中的体现 |
|------|---------------------|
| **互助盘** | ETH/LSD质押赚租金，泡沫源自机会成本，抽水来自TVL |
| **分红盘** | 下游"共享安全性"协议需要质押，提供分红锁定 |
| **拆分盘** | 不断拆分出更多需要安全的分红盘协议 |

### 3.5 Restaking的组合逻辑

**互助盘抽水最大化TVL**：
> "Restaking互助盘抽水来自于TVL，因此最大化TVL是第一优先"

**用分红盘的【锁】实现**：
- Restaking下游是"共享安全性"协议
- 对于安全的定义 = 质押量高
- 质押要分红 → 这些协议本质是分红盘

**用拆分盘的【销】扩展**：
- 分红盘越多，Restaking TVL越高
- 币价涨、APY涨，用户愿意质押
- 新分红盘往往是低流动性资产，很好拉盘

### 3.6 从ETH庄家视角看

**以太坊POS是质押分红盘**：需要通过LSD释放ETH筹码拉拆分盘

**熊市问题**：
- 拆分标的跑不赢Beta
- 新项目少拆分减缓
- ETH变回抛压

**Restaking的作用**：
> "需要用Restaking提供足够的市场beta，让ETH筹码锁住不砸盘"

### 3.7 飞轮效应

> "左脚踩右脚"的飞轮：

```
在牛初释放更多"基础设施"分红盘到Restaking客户端
    ↓
提高Restaking的APY
    ↓
促使大量$ETH被锁定
    ↓
更容易拉升ETH
    ↓
更低成本让整个以太板块形成正向飞轮
    ↓
锁ETH → 拉飞ETH → 大量发项目需求Restaking质押 → APY预期上升 → 更多锁定
```

### 3.8 代价与问题

**对"基础设施"项目方的问题**：
> "如果'共享安全性'的代价是你做个分红盘用户入金都不在你手上，你的原生代币甚至'治理'价值都要让一部分给ETH，而且万一Eigenlayer出事你会跟着挂。作为一个盘主，你真的愿意吗？"

**中心化代价**：
> "这样的模型，需要ETH庄家集团-Restaking提供方-'共享安全'需求方利益一致，甚至是一家人"

> "调度这么多利益相关方，更需要非常好的可组合性，需要一个协调一致的'中枢共识'来指挥"

> "说白了，整个生态会'不得不'变得更中心化"

## 四、以太坊生态的三盘结合

### 4.1 基本盘：分红盘

- POW时代：矿机分红盘
- POS时代：质押分红盘
- 通胀产出在ICO前只能成为卖压

### 4.2 拆分盘时代：ICO

ICO出现后，开启以太的拆分盘时代

**拆分逻辑**：
- ETH/BSC发币需要项目方投入运营购买注意力（如空投）
- 获取系统流动性
- 不同于币安首发的"系统级抽血"

### 4.3 存量抛售问题的解决

**ICO时代问题**：
- 存量抛售严重
- 项目方募资后砸盘
- 18年价格最低跌破100U

**DeFi后的解决**：
- 链上应用发展
- 涉及三盘结合

### 4.4 Vitalik的叙事控制

> "从拆分盘角度来说，ICO时代无序拆分给Vitalik带来巨大阴影"

**应对方式**：
> "通过叙事以及以太内圈白手套，给拆分立规矩"

**叙事演进**：
```
DeFi叙事 → NFT/SBT非金融应用 → LSD → Layer2 → Restaking
```

**效果**：每个叙事的核心位置都非常有限，**控制了拆分的速率**

### 4.5 反身性问题

**负面后果**：
- to V创业成主流
- 项目哲学愈发同质化
- 正统性=高估值，野路子/低流动性不再
- 拆分速率不足
- Beta被BTC、Solana碾压

## 五、防崩盘的三盘结合

### 5.1 分红盘的防崩盘调整

当分红盘面临崩盘（筹码结构分散、失去上涨动能）：

| 策略 | 盘型结合 |
|------|----------|
| 降低算力或封号 | 调整分红盘参数 |
| 套娃新的分红盘或拆分盘 | 三盘结合 |
| 找到长期流动性（上所） | 引入外部流动性 |

### 5.2 结合原则

**何时需要结合**：
- 市场流动性匮乏
- 筹码结构变得更为分散
- 矿机盘进入第二阶段

**结合目的**：
- 分担抛压
- 延长生命周期
- 扩大适用范围

## 六、三盘结合设计原则

### 6.1 识别单一盘型的局限

| 盘型 | 局限 |
|------|------|
| 分红盘 | 需要持续入金，筹码分散后失去动能 |
| 互助盘 | 难以资产发行，数值调整空间小 |
| 拆分盘 | 需要持续新标的，受Beta影响大 |

### 6.2 结合策略

| 主盘型 | 结合策略 |
|--------|----------|
| 分红盘为主 | + 拆分盘消耗质押筹码 + 互助盘提速 |
| 互助盘为主 | + 分红盘锁仓 + 拆分盘资产发行 |
| 拆分盘为主 | + 分红盘提供基础权益 + 互助盘加速流动 |

### 6.3 公链盘设计思路

> "善于开盘的你，是不是已经知道下一个公链盘应该怎么做了？"

**要素**：
1. 质押分红盘作为基本盘
2. 可组合性良好的Restaking类组件
3. 叙事控制下的有序拆分
4. 利益相关方的协调机制

## 七、可组合性的技术实现

### 7.1 开放式协议

Crypto可组合性的技术基础：
- 智能合约的开放性
- 协议间的互操作性
- 无需许可的接入

### 7.2 Token层面的可组合性

**"功能在Token上"的逻辑**：
- 将复杂逻辑塞进Token
- 减少DAPP开发
- 提高可复用性

**工具**：
- Tokenscript/EIP5169
- ERC404等新标准

### 7.3 协议层面的可组合性

**案例**：
- Uniswap作为通用流动性层
- Restaking作为安全租赁层
- 积分系统作为获客层

## 八、三盘结合检查清单

### 8.1 设计阶段

| 检查项 | 问题 |
|--------|------|
| 主盘型选择 | 核心商业模式是分红、互助还是拆分？ |
| 辅助盘型 | 需要什么机制补足主盘型的局限？ |
| 接口设计 | 不同盘型模块如何对接？ |
| 利益分配 | 各方如何从组合中获益？ |

### 8.2 运营阶段

| 检查项 | 问题 |
|--------|------|
| 平衡监控 | 各盘型模块是否平衡运行？ |
| 崩盘预警 | 哪个模块最可能先崩？ |
| 调整空间 | 是否预留了参数调整能力？ |
| 退出路径 | 组合崩溃时如何有序退出？ |
